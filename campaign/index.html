Access Control — Deny by Default

CRA §2d Ensure protection against unauthorized access by appropriate control mechanisms
Program.cs — Authorization configuration
builder.Services.AddAuthorization(options =>
{
-   // No fallback means new endpoints are open by default
-   // Forgot [Authorize]? Open to everyone.
+   // Deny by default: everything requires auth
+   options.FallbackPolicy = new AuthorizationPolicyBuilder()
+       .RequireAuthenticatedUser()
+       .Build();
});
OrdersController.cs — IDOR vulnerability
+// Attack: GET /api/orders/42 returns my order
+//         GET /api/orders/43 returns someone else's (IDOR)
[HttpGet("{id}")]
public IActionResult GetOrder(int id)
{
    var order = db.Orders.Find(id);
-   return Ok(order);
+   if (order == null) return NotFound();
+   if (order.UserId != User.FindFirstValue(ClaimTypes.NameIdentifier))
+       return Forbid();
+   return Ok(order);
}
Risk

Without a fallback policy, every new endpoint is open by default. Forget [Authorize] once and you expose customer data. IDOR lets user A see user B's orders by changing the ID in the URL.

Fix

3 lines of middleware + 2 lines per endpoint.

Secure Defaults — Fail Securely

CRA §2b Made available with a secure configuration by default
CRA §2k Minimize impact of an incident through appropriate mechanisms
AdminController.cs — Dashboard access
[HttpGet("admin/dashboard")]
public IActionResult GetDashboard()
{
+   var userId = User.FindFirstValue(ClaimTypes.NameIdentifier);
-   bool isAdmin = true;
+   bool isAdmin = false;
    try
    {
        isAdmin = _authService.IsUserInRole(userId, "Administrator");
    }
    catch (Exception ex)
    {
-       return Content(ex.ToString()); // leaks stacktrace
+       _logger.LogWarning(ex, "Admin check failed for {User}", userId);
+       // isAdmin stays false = fail secure
    }

-   if (isAdmin) return View("Dashboard");
-   return Unauthorized();
+   if (!isAdmin) return Forbid();
+   return View("Dashboard");
}
Risk

isAdmin = true as default means any exception grants admin access. ex.ToString() leaks the stacktrace directly to the attacker.

Fix

Flip the default to false, log the error server-side. The system now fails closed instead of open.

Supply Chain — SBOM & Dependency Scanning

CRA §2a Made available without known exploitable vulnerabilities
CRA §2c Ensure vulnerabilities can be addressed through security updates
build.yml — CI/CD pipeline
steps:
  - name: Build
    run: dotnet build

+  - name: Dependency vulnerability scan
+    run: |
+      dotnet list package --vulnerable --include-transitive 2>&1 | tee vuln.txt
+      if grep -q "has the following vulnerable packages" vuln.txt; then exit 1; fi
+
+  - name: Generate SBOM
+    run: dotnet CycloneDX App.sln -j -o .
+
+  - name: Upload SBOM to Dependency-Track
+    run: |
+      curl -X POST "${{ secrets.DT_URL }}/api/v1/bom" \
+        -H "X-Api-Key: ${{ secrets.DT_API_KEY }}" \
+        -F "projectName=App" -F "autoCreate=true" -F "bom=@bom.json"

  - name: Deploy
    run: dotnet publish
Risk

Most of a modern codebase is dependencies. Without scanning, we ship other people's vulnerabilities.

Fix

3 pipeline steps: scan, generate SBOM, upload. The build now fails on known vulnerabilities.

Confidentiality & Data Integrity

CRA §2e Protect confidentiality by encrypting relevant data at rest or in transit
CRA §2f Protect integrity of data against unauthorized modification

§2e Encrypt data in transit:

Program.cs — HTTPS enforcement
var app = builder.Build();

-// HTTP allows traffic to be intercepted on the network
-app.MapControllers();
+app.UseHttpsRedirection();
+app.UseHsts();
+app.MapControllers();

§2e Protect stored credentials:

UserService.cs — Password storage
-db.Store(new User { Password = request.Password });
+var hash = BCrypt.Net.BCrypt.HashPassword(request.Password, workFactor: 12);
+db.Store(new User { PasswordHash = hash });

§2f Protect integrity (SQL injection):

OrderRepository.cs — Database query
-var query = "SELECT * FROM users WHERE id=" + userId;
-var result = db.ExecuteRaw(query);
+var result = db.Users
+    .Where(u => u.Id == userId)
+    .FirstOrDefault();
+// EF Core parameterizes automatically, preventing SQL injection

§2e Remove secrets from source code:

appsettings.json + Program.cs — Connection string
-"ConnectionString": "Server=db;Database=app;User=sa;Password=hunter2"
+"ConnectionStrings": {
+  "Default": "" // injected via Key Vault or environment at runtime
+}
Risk

Plaintext passwords: one breach exposes every user. String concatenation in SQL: attacker reads the entire database. Hardcoded credentials in git: permanently exposed. HTTP without TLS: all traffic visible on the wire.

Fix

One line HTTPS, one line bcrypt, one line LINQ, secrets out of code. None takes more than 5 minutes.

Attack Surface Reduction

CRA §2j Designed to limit attack surfaces, including external interfaces
CRA §2h Protect availability of essential functions, even after an incident
Program.cs — Rate limiting, debug endpoints, security headers
+// Rate limiting against DoS
+builder.Services.AddRateLimiter(options =>
+{
+    options.AddFixedWindowLimiter("default", opt =>
+    {
+        opt.PermitLimit = 100;
+        opt.Window = TimeSpan.FromMinutes(1);
+    });
+});

var app = builder.Build();
+app.UseRateLimiter();

-// Debug endpoints active in production
-app.MapGet("/debug/config", () => Results.Json(config));
-app.MapGet("/debug/env", () => Results.Json(Environment.GetEnvironmentVariables()));

+if (app.Environment.IsDevelopment())
+{
+    app.MapGet("/debug/config", () => Results.Json(config));
+}

+// Security headers (CRA §2j)
+app.Use(async (context, next) =>
+{
+    var h = context.Response.Headers;
+    h["X-Content-Type-Options"] = "nosniff";
+    h["X-Frame-Options"] = "DENY";
+    h["Strict-Transport-Security"] = "max-age=31536000; includeSubDomains";
+    await next();
+});
Risk

Debug endpoints in production leak config and environment variables. No rate limiting means free brute-force and DDoS. Missing security headers allow clickjacking and MIME sniffing.

Fix

Debug behind environment check, rate limiter added, security headers set.

Security Logging & Incident Readiness

CRA §2l Provide security information by recording and monitoring relevant activities
NIS2 24-hour early warning for incidents
AuthController.cs — Login endpoint
[HttpPost("login")]
public IActionResult Login(LoginDto dto)
{
    var user = authService.Authenticate(dto.Username, dto.Password);
-   if (user == null) return Unauthorized();
+   if (user == null)
+   {
+       _logger.LogWarning(
+           "Failed login. User: {Username}, IP: {IP}, UA: {UserAgent}",
+           dto.Username,
+           HttpContext.Connection.RemoteIpAddress,
+           Request.Headers.UserAgent.ToString());
+       return Unauthorized("Invalid username or password");
+   }
+
+   _logger.LogInformation(
+       "Successful login. User: {UserId}, IP: {IP}",
+       user.Id,
+       HttpContext.Connection.RemoteIpAddress);
    return Ok(GenerateToken(user));
}
Program.cs — Structured logging + metrics
-builder.Host.UseSerilog();
+builder.Host.UseSerilog((ctx, config) => config
+    .MinimumLevel.Information()
+    .WriteTo.Console()
+    .WriteTo.Seq("http://seq:5341")
+    .Enrich.WithProperty("App", "MyApp"));
+
+// Metrics for Prometheus — enables alerting
+builder.Services.AddOpenTelemetry()
+    .WithMetrics(m => m
+        .AddAspNetCoreInstrumentation()
+        .AddPrometheusExporter());

var app = builder.Build();
+app.MapPrometheusScrapingEndpoint();
Risk

return Unauthorized() without logging means we're blind. When brute-force hits, we have no data. NIS2 requires a 24-hour early warning — without structured logging and alerting we can't comply with that.

Fix

Structured logging with context (who, from where, when) + metrics exposed to monitoring. When an incident happens, we have the data to react within 24 hours.
